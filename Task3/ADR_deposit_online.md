### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| NFR-01 | **Доступность 24/7, SLA 99,9%.** Все онлайн-сервисы (интернет-банк, сайт) должны быть доступны круглосуточно с доступностью не менее 99,9%. |
| NFR-02 | **Отказоустойчивость ЦОД.** Сервисы должны работать при сбое основного ЦОД с переключением на резервный. Новые микросервисы должны быть развёрнуты в обоих ЦОД. |
| NFR-03 | **Производительность.** Отклик по операциям — миллисекунды. Справочные данные (включая ставки) должны загружаться менее 1 секунды. |
| NFR-04 | **Защита АБС от онлайн-нагрузки.** Прямая интеграция интернет-банка с API АБС запрещена. БД АБС перегружена, новые онлайн-процессы не должны создавать дополнительную нагрузку на неё. Использовать асинхронное взаимодействие через Kafka. |
| NFR-05 | **Горизонтальное масштабирование.** Новые сервисы должны поддерживать горизонтальное масштабирование и балансировку нагрузки между серверами и ЦОД. АБС масштабируется только вертикально. |
| NFR-06 | **Шифрование данных.** Все каналы передачи данных (сайт, интернет-банк) должны использовать TLS/HTTPS. |
| NFR-07 | **Использование существующих технологий.** Приоритет на MS SQL, Oracle, .NET, Java, Python. Совместимость с существующими платформами. Внутренняя экспертиза по новым технологиям обязательна. |
| NFR-08 | **Kafka как брокер сообщений.** Для асинхронного взаимодействия использовать Apache Kafka. Текущая платформа ИБ (ASP.NET MVC 4.5) несовместима с Kafka — новые компоненты должны быть на совместимом стеке. |
| NFR-09 | **Микросервисная архитектура.** Новый функционал депозитов реализовать в виде микросервисов, отдельных от монолита интернет-банка. Это снимает зависимость от подрядчика ядра ИБ. |
| NFR-10 | **Независимость от подрядчика ядра ИБ.** SMS-функционал реализован в ядре ИБ и требует доработки подрядчика. Новый SMS-сервис реализовать силами команды банка. |
| NFR-11 | **Разделение доступа.** Сотрудники депозитов и кредитов не должны иметь доступа к данным друг друга. Сервис управления ставками должен обеспечить ролевое разграничение. |
| NFR-12 | **Документация.** Необходимо предусмотреть техническую документацию для новых сервисов и API для дальнейшего расширения. |
| NFR-13 | **Соответствие дизайн-системе.** Все UI-компоненты интернет-банка и сайта должны соответствовать принятой в компании системе дизайна. |

### **Решение**

#### Общий подход

Для реализации MVP открытия депозитов онлайн предлагается архитектура, основанная на следующих принципах:

1. **Микросервисная декомпозиция нового функционала** — вместо доработки монолита интернет-банка создаются новые микросервисы, работающие рядом с существующим монолитом.
2. **Асинхронное взаимодействие через Kafka** — защищает АБС от пиковых онлайн-нагрузок и обеспечивает надёжную доставку сообщений.
3. **Сервис интеграции депозитов в АБС** — новый API-слой поверх АБС, который инкапсулирует работу с депозитами и ставками.
4. **Централизованное управление ставками** — замена Excel-процесса на сервис управления ставками в АБС с API-доступом.

#### Логика принятия решений

1. **Почему микросервисы, а не доработка монолита ИБ:**
   - Монолит на ASP.NET MVC 4.5 / .NET Framework 4.5 — устаревший стек, несовместимый с Kafka.
   - Обновление ядра ИБ привязано к подрядчику — долго и дорого.
   - SMS-функционал в ядре требует доработки подрядчика.
   - Микросервисы на .NET Core совместимы с Kafka и масштабируются горизонтально.

2. **Почему Java Spring Boot для сервисов АБС:**
   - В команде АБС есть Java-специалисты (работают с системой кол-центра).
   - Java Spring Boot — зрелый стек для enterprise-сервисов.
   - Совместимость с Oracle DB и Kafka.
   - Система кол-центра уже построена на Java Spring Boot — есть экспертиза и инфраструктура.

3. **Почему Kafka, а не синхронные API:**
   - АБС масштабируется только вертикально, БД перегружена.
   - Kafka обеспечивает буферизацию нагрузки — пиковые потоки заявок не перегрузят АБС.
   - Гарантированная доставка сообщений (at-least-once) — заявки не теряются.
   - Задание рекомендует использовать Kafka на перспективу.

4. **Почему Rate Management Service в АБС, а не отдельная система:**
   - Ставки напрямую связаны с данными АБС (кредитный риск, рефинансирование ЦБ).
   - Бэк-офис депозитов и кредитов уже работает в АБС — минимальные изменения в процессе.
   - Ролевое разграничение доступа решается на уровне API сервиса.

5. **Почему отдельная Deposit DB для микросервисов ИБ:**
   - Разделение данных между монолитом и новыми сервисами (Database per Service).
   - Кеширование ставок в Deposit DB — быстрый отклик без нагрузки на АБС.
   - Независимое масштабирование.

### **Альтернативы**

| № | Альтернатива | Причина отказа |
|---|-------------|----------------|
| 1 | **Доработка монолита ИБ** вместо микросервисов | ASP.NET MVC 4.5 несовместим с Kafka. Зависимость от подрядчика для изменения ядра. SMS-функционал требует доработки подрядчика. Нет горизонтального масштабирования. |
| 2 | **Прямая интеграция ИБ ↔ АБС через REST API** вместо Kafka | БД АБС перегружена, синхронные онлайн-запросы могут привести к отказу. Отсутствие буферизации нагрузки. АБС масштабируется только вертикально. |
| 3 | **Сохранить интеграцию ИБ ↔ АБС через БД** (текущий подход) | Жёсткая связанность систем. Не обеспечивает асинхронность и буферизацию. Дополнительная нагрузка на перегруженную БД Oracle. |
| 4 | **Хранение ставок в отдельном сервисе** вне АБС | Дублирование данных, необходимых для расчёта ставок (кредитный риск в АБС). Сложность синхронизации. Бэк-офис и так работает в АБС. |
| 5 | **RabbitMQ вместо Kafka** как брокер сообщений | Бизнес рекомендует Kafka на перспективу. Kafka лучше подходит для event-driven архитектуры и обработки потоков данных. Kafka обеспечивает повторное чтение сообщений при сбоях. |
| 6 | **Реализация всех новых сервисов на Java** (единый стек) | В команде ИБ основная экспертиза — .NET. Использование .NET Core для ИБ-сервисов снижает порог входа и ускоряет разработку. Java используется там, где есть Java-эксперты (команда АБС). |

**Недостатки, ограничения, риски**

| № | Категория | Описание | Митигация |
|---|----------|----------|-----------|
| 1 | **Риск** | Внедрение Kafka увеличивает сложность инфраструктуры. Нет внутренней экспертизы по Kafka. | Привлечение внешнего эксперта для настройки. Обучение IT-команды. Kafka — популярная технология, специалисты доступны на рынке. |
| 2 | **Ограничение** | На этапе MVP бэк-офис обрабатывает заявки вручную. Скорость открытия депозита зависит от оператора. | Это осознанное ограничение MVP. На следующем этапе процесс будет автоматизирован. |
| 3 | **Риск** | Rate Management Service создаёт дополнительную нагрузку на Oracle БД АБС. | Использование отдельной схемы в Oracle. Кеширование ставок в Deposit Service ИБ. Мониторинг нагрузки на БД. |
| 4 | **Недостаток** | Два технологических стека (.NET Core для ИБ, Java для АБС) — увеличенная стоимость поддержки. | Стеки соответствуют экспертизе имеющихся команд. Unified API contracts через OpenAPI/Swagger. |
| 5 | **Риск** | Eventual consistency — заявка попадает в АБС с задержкой через Kafka. Клиент не видит моментального результата. | В MVP это приемлемо (обработка менеджером). Статус заявки отображается в ИБ, уведомления через SMS. |
| 6 | **Ограничение** | Новые клиенты с сайта обязаны пройти идентификацию в отделении (KYC). Онлайн-открытие депозита доступно только существующим клиентам ИБ. | Регуляторное ограничение, не может быть снято архитектурным решением. |
| 7 | **Риск** | Команда ИБ (10 внутренних + 10 подрядчик) может не справиться с параллельной поддержкой монолита и разработкой микросервисов. | Чёткое разделение ответственности: подрядчик поддерживает монолит, внутренняя команда разрабатывает микросервисы. Привлечение дополнительных ресурсов при необходимости. |
| 8 | **Недостаток** | API Gateway добавляет дополнительный hop для всех запросов — незначительное увеличение латентности. | Использование легковесного API Gateway (Ocelot для .NET). Развёртывание на тех же серверах для минимизации сетевых задержек. |
