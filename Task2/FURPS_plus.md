## Functionality — Функциональные требования

### Депозитное направление (сайт)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-D-01 | На сайте новый клиент видит список доступных депозитов с актуальными ставками | Необходим механизм публикации актуальных ставок из внутренних систем банка на сайт |
| F-D-02 | Клиент может подать заявку на депозит, оставив номер телефона и Ф. И. О. | Минимальная форма для лидогенерации |
| F-D-03 | После подачи заявки клиенту звонит менеджер кол-центра, который видит заявку в системе кол-центра и может предложить особые условия | Требуется интеграция: Сайт → Система кол-центра |
| F-D-04 | После звонка новому клиенту необходимо прийти в отделение для идентификации (KYC) | Регуляторное требование — банки не могут принимать новых клиентов без личной проверки документов |

### Депозитное направление (интернет-банк)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-D-05 | В интернет-банке клиент видит список доступных депозитов с актуальными ставками | Требуется источник актуальных ставок (сейчас ведутся только в Excel) |
| F-D-06 | Клиент видит персонализированные ставки лично для него | Требуется механизм расчёта/получения персональной ставки для клиента |
| F-D-07 | Клиент может подать заявку на депозит, указав счёт и сумму | Новый функционал в интернет-банке |
| F-D-08 | Операция подачи заявки подтверждается СМС-кодом | Функционал SMS реализован в ядре системы, но требует доработки подрядчика. Лучше реализовать силами команды банка |
| F-D-09 | Клиент получает СМС-уведомление после подтверждения ставки | Используется существующий СМС-шлюз через телеком-оператора |
| F-D-10 | Клиент получает СМС-уведомление после открытия депозита | Используется существующий СМС-шлюз через телеком-оператора |

### Депозитное направление (бэк-офис / MVP)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-D-11 | На этапе MVP заявку на депозит обрабатывает менеджер бэк-офиса, подтверждая условия в АБС | Промежуточное решение до полной автоматизации |
| F-D-12 | Ставки по депозитам должны вестись в информационной системе (не только в Excel) | Сейчас ставки ведутся только в XLS-файлах. Возможно реализовать в АБС |
| F-D-13 | С данными по ставкам должны работать сотрудники бэк-офиса депозитов и бэк-офиса кредитов | При этом сотрудники депозитов и кредитов не должны иметь доступ к данным друг друга (требование безопасности из описания процесса) |
| F-D-14 | Процесс согласования ставки по заявке должен быть автоматизирован в АБС | Текущий процесс включает ручной обмен email и Excel между бэк-офисами |

### Кредитное направление (сайт)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-K-01 | На сайте клиент видит предложение открыть кредит с информацией о ставке | Полная информация об условиях кредита хранится в Кредитном конвейере |
| F-K-02 | Клиент может подать заявку на кредит, оставив номер телефона и Ф. И. О. | Минимальная форма заявки |
| F-K-03 | Если клиент заполнит паспортные данные — ему сразу выдаётся решение об одобрении/неодобрении кредита | Для нового клиента используются только данные из Бюро кредитных историй. Требуется онлайн-интеграция Сайт → Скоринг → БКИ |
| F-K-04 | После подачи заявки клиенту приходит СМС о необходимости прийти в отделение для получения кредита и идентификации | Регуляторное требование — идентификация нового клиента |

### Кредитное направление (интернет-банк)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-K-05 | В интернет-банке клиент видит список доступных кредитов с актуальными условиями | Аналогично сайту, данные из Кредитного конвейера |
| F-K-06 | Клиенту может выводиться предодобренное предложение по кредиту (предрасчитанный скоринг и приемлемая сумма) | Предполагает предварительный скоринг клиента. При этом систему скоринга нельзя нагружать дополнительными процессами в рабочие часы |
| F-K-07 | После подачи заявки по предодобренному предложению скоринг проводится повторно по упрощённой процедуре | Такие заявки обрабатываются быстрее сотрудниками |
| F-K-08 | Клиент указывает счёт зачисления средств и данные для подачи заявки и отправляет её в банк | Новый функционал в интернет-банке |
| F-K-09 | Операция подачи заявки на кредит подтверждается СМС-кодом | Аналогично депозитной заявке, SMS-подтверждение |
| F-K-10 | Информирование клиента по СМС об изменении статуса заявки на кредит | Новое уведомление через существующий СМС-шлюз |

### Кредитное направление (отделение)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-K-11 | В отделении менеджер может подать заявку на кредит через АБС | Существующий процесс сохраняется |
| F-K-12 | Если клиент получил предварительное одобрение на сайте, менеджер должен продолжить работу с той же заявкой | Требуется сквозная идентификация заявки между каналами (Сайт → АБС → Кредитный конвейер) |

### Кредитное направление (бэк-офис)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| F-K-13 | Заявки по предодобренным предложениям обрабатываются сотрудниками в течение того же рабочего дня | Сейчас заявки попадают в Кредитный конвейер из АБС раз в сутки. Ускорить обмен через БД невозможно → нужен альтернативный канал передачи |
| F-K-14 | Предрасчёт скорингов не должен нагружать систему скоринга дополнительно в рабочие часы | Система скоринга испытывает повышенную нагрузку в рабочие часы |

## Usability — Удобство использования

| ID | Требование | Комментарий |
|----|-----------|-------------|
| U-01 | Интерфейс работы должен быть максимально удобным для клиента | Целевая аудитория расширяется на молодых клиентов — UX критичен для привлечения |
| U-02 | Необходимо придерживаться системы дизайна для приложений, принятой в компании | Единообразие UI/UX между каналами |
| U-03 | Пользователи должны видеть привычные цвета и брендированные элементы | Brand identity в цифровых каналах |
| U-04 | Интерфейс должен быть понятен для людей старшего возраста (текущая ЦА банка) | Основная категория клиентов — люди старшего возраста. При переходе в онлайн важно не создать барьеров для существующих клиентов |

## Reliability — Надёжность

| ID | Требование | Комментарий |
|----|-----------|-------------|
| R-01 | Все сервисы должны работать 24/7 | Онлайн-каналы требуют круглосуточной доступности |
| R-02 | Доступность сервисов — 99,9% | ~8,76 часов допустимого простоя в год |
| R-03 | В случае сбоев в ЦОД сервисы интернет-банка должны быть доступны и выдерживать требуемую нагрузку | Текущая архитектура — один ЦОД + резервный. Команда ИБ утверждает, что требования пока невыполнимы |
| R-04 | Необходимо обеспечить переключение на резервный ЦОД в случае сбоя основного | Резервный ЦОД уже есть, нужно обеспечить механизм failover |
| R-05 | Необходимо избежать прямой работы интернет-банка с API АБС в новом процессе | БД АБС перегружена; прямое взаимодействие может поставить под угрозу работоспособность банка |
| R-06 | Кредитный конвейер и система скоринга должны поддерживать работу 24/7 | Уже реализовано — системы разработаны с учётом этого требования |
| R-07 | АБС должна оставаться работоспособной при увеличении онлайн-нагрузки | АБС — критическая система банка: учёт операций, бухгалтерия. Её падение парализует все процессы, включая обслуживание в отделениях |
| R-08 | Новые компоненты не должны создавать единую точку отказа (SPOF) | При проектировании микросервисов для депозитного процесса необходимо обеспечить отказоустойчивость каждого компонента |

## Performance — Производительность

| ID | Требование | Комментарий |
|----|-----------|-------------|
| P-01 | Отклик по всем операциям в интернет-банке должен быть максимально быстрым — миллисекунды | Бизнес-ожидание по скорости отклика |
| P-02 | Справочные данные должны загружаться менее чем за секунду (сейчас загружаются дольше) | Текущая проблема при проведении платежей. Возможно, нужно кеширование справочников |
| P-03 | Необходимо обеспечить горизонтальное масштабирование и распределение запросов между серверами, приложениями и ЦОД | Подготовка к росту нагрузки при привлечении новых клиентов |
| P-04 | АБС может масштабироваться только вертикально из-за своей базы данных (Oracle) | Ограничение архитектуры АБС. Нужно минимизировать нагрузку на АБС от онлайн-каналов |
| P-05 | Ускорить обмен данными между АБС и Кредитным конвейером (чаще чем раз в сутки через БД) невозможно | Техническое ограничение. Для обработки кредитных заявок в тот же день нужен альтернативный канал доставки |
| P-06 | Систему скоринга нельзя нагружать дополнительными процессами предрасчёта скорингов в рабочие часы | Система испытывает повышенную нагрузку в рабочие часы |
| P-07 | Кредитный конвейер и система скоринга поддерживают горизонтальное масштабирование | Уже реализовано, можно использовать для увеличения пропускной способности |
| P-08 | При росте числа онлайн-клиентов нагрузка на сайт и ИБ значительно возрастёт | Банк планирует маркетинговую кампанию для привлечения молодых клиентов. Нужно заложить запас по нагрузке |

## Supportability — Поддерживаемость

| ID | Требование | Комментарий |
|----|-----------|-------------|
| S-01 | При доработках максимально использовать технологии, которые уже есть в банке (MS SQL, Oracle) | Снижение рисков и стоимости за счёт использования имеющейся экспертизы |
| S-02 | Новые технологии должны быть совместимы с существующими платформами разработки | .NET, Java, Python, PHP — стеки, имеющиеся в банке |
| S-03 | Внутри банка должна быть экспертиза по используемым технологиям (хотя бы на уровне языков программирования) | В банке есть специалисты по .NET, Java, Python, PHP, Delphi/PL-SQL |
| S-04 | Для очередей сообщений использовать Kafka на перспективу | Текущая платформа интернет-банка (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka |
| S-05 | Рассмотреть перевод интернет-банка на микросервисную архитектуру (в рамках задачи открытия депозитов) | Позволит отвязаться от ядра подрядчика и внедрять Kafka, горизонтальное масштабирование |
| S-06 | Предусмотреть разработку документации для дальнейшего расширения системы | Критично для онбординга новых разработчиков и развития архитектуры |
| S-07 | SMS-функционал реализован в ядре ИБ и требует доработки подрядчика — лучше реализовать силами банка | Зависимость от подрядчика увеличивает сроки и стоимость. Реализация SMS через отдельный микросервис — возможная альтернатива |
| S-08 | Обновление ядра интернет-банка привязано к подрядчику | Команда банка может вносить изменения, но ядро — только подрядчик. Микросервисный подход позволит обходить это ограничение |
| S-09 | Система кол-центра имеет возможности для расширения; в команде АБС есть Java-специалисты | Можно использовать систему кол-центра (Java Spring Boot, микросервисы) как платформу для новых интеграций |
| S-10 | Кредитный конвейер — коробочное решение на Camunda BPM (Java), частично дорабатываемое подрядчиком и IT-отделом | IT-отдел может менять обработчики и описания бизнес-процессов самостоятельно |

## Ограничения проектирования (Design Constraints)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| DC-01 | MVP депозитов: бэк-офис обрабатывает заявки по старому процессу, через 6 месяцев | Промежуточное состояние — не нужна полная автоматизация на этапе MVP |
| DC-02 | Финальное состояние (через год): депозиты открываются полностью автоматически без участия бэк-офиса | Сокращение операционных издержек |
| DC-03 | Новые клиенты обязаны пройти идентификацию в отделении | Регуляторное ограничение (KYC) |
| DC-04 | Сотрудники депозитов и кредитов не должны иметь доступа к данным друг друга | Требование информационной безопасности. Влияет на архитектуру разграничения доступа к данным о ставках |
| DC-05 | Кредитная заявка должна проходить сквозь каналы: предодобрение на сайте должно быть доступно менеджеру в отделении | Омниканальность — единая заявка через все каналы взаимодействия |
| DC-06 | Решение по кредиту клиент должен получить в течение одного дня (если данных достаточно) | Как для заявок с сайта, так и из интернет-банка |

## Ограничения реализации (Implementation Constraints)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| IC-01 | Интернет-банк — монолит на ASP.NET MVC 4.5 / .NET Framework 4.5 / MS SQL | Устаревший стек, несовместим с Kafka. Миграция на микросервисы — стратегическое решение |
| IC-02 | АБС — десктопный клиент на Delphi, логика в PL-SQL на Oracle | Масштабируется только вертикально. Интеграция через БД — основной паттерн |
| IC-03 | Интернет-банк интегрирован с АБС напрямую через БД | Жёсткая связанность, риск при изменениях. Для нового процесса лучше использовать другой тип интеграции |
| IC-04 | Обмен АБС ↔ Кредитный конвейер — раз в сутки через интеграцию по БД | Ускорить невозможно. Для обработки заявок день-в-день нужен новый канал (например, через API или Kafka) |
| IC-05 | Система кол-центра — микросервисная (React.js, Java Spring Boot, PostgreSQL), на платформе подрядчика | Расширяема, есть внутренняя экспертиза по Java |
| IC-06 | Сайт — собственная разработка на PHP и React.js | Полный контроль над доработками |
| IC-07 | Система скоринга — Python, Flask REST API, PostgreSQL | Разрабатывается внутри банка, REST API — удобен для интеграции |
| IC-08 | Кредитный конвейер — Camunda BPM, Java, Oracle | Частично подрядчик, частично внутренняя доработка |
| IC-09 | Данные в систему скоринга из АБС и Кредитного конвейера передаются раз в сутки через БД | Ограничение актуальности данных для скоринга |
| IC-10 | Внутренняя IT-команда: 10 чел. (ИБ) + 10 чел. подрядчик (ИБ) + 20 чел. (АБС) + Java-специалисты + Python-разработчики | Ограниченный ресурс для параллельной разработки |

## Требования к интерфейсам (Interface Requirements)

| ID | Требование | Комментарий |
|----|-----------|-------------|
| IR-01 | Данные с сайта должны защищаться механизмом шифрования трафика (TLS/HTTPS) | С сайта передаётся чувствительная информация (ФИО, телефон, паспортные данные) |
| IR-02 | Данные в интернет-банке при передаче должны быть защищены механизмом шифрования | Финансовые данные клиентов — банковская тайна |
| IR-03 | Бюро кредитных историй предоставляет данные через REST API | Внешний API, онлайн-запросы |
| IR-04 | Система скоринга предоставляет REST API на Flask | Внутренний API для оценки кредитоспособности |
| IR-05 | Партнёрский кол-центр — внешняя система, интеграция через скрипты от бизнес-представителей | Ограниченные возможности автоматической интеграции |
| IR-06 | СМС-шлюз поддерживается IT-отделом, взаимодействует с внешним телеком-оператором | Единый канал SMS-уведомлений |
| IR-07 | Новые микросервисы должны предоставлять API, совместимые с существующими интеграциями | Унификация взаимодействия между системами снижает стоимость сопровождения |

## Физические ограничения (Physical Constraints)

| ID | Требование | Комментарий |
|----|-----------|-------------|
