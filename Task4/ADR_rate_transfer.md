### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| NFR-01 | **Актуальность ставок.** Ставки обновляются ежедневно. Система кол-центра должна получать ставки в режиме реального времени через API. Партнёрский кол-центр получает ставки с задержкой не более нескольких часов (файловый обмен). |
| NFR-02 | **Безопасность SFTP.** SFTP-сервер должен быть защищён: аутентификация по ключам SSH, ограничение IP-адресов партнёра, шифрование данных при передаче. |
| NFR-03 | **Доступность API ставок.** Rate Management Service API должен быть доступен 24/7 с SLA 99,9%, так как кол-центр работает круглосуточно. |
| NFR-04 | **Минимальные изменения в существующих системах.** Система кол-центра — платформа подрядчика, но расширяема (микросервисная архитектура, Java Spring Boot). Изменения должны быть минимальны и реализуемы силами Java-специалистов банка. |
| NFR-05 | **Формат файлов.** Файлы со ставками для партнёра — в формате CSV или JSON. Формат согласовывается с партнёром. |
| NFR-06 | **Мониторинг.** Необходимо отслеживать доставку файлов на SFTP и доступность API ставок. Алертинг при сбоях. |
| NFR-07 | **Использование существующих технологий.** Java Spring Boot для новых сервисов на стороне АБС (есть экспертиза). SFTP — стандартный протокол, не требует специфических технологий. |

### **Решение**

#### Общий подход

Решение основано на Rate Management Service, спроектированном в рамках Task 3 для MVP депозитов. Этот сервис уже предоставляет REST API для управления и получения ставок. Для нового кейса необходимо:

1. **Интеграция системы кол-центра с API ставок** — система кол-центра (Java Spring Boot, микросервисная) получает ставки через REST API Rate Management Service.
2. **SFTP-сервер для партнёрского кол-центра** — новый инфраструктурный компонент для файлового обмена.
3. **Rate Export Service** — новый компонент в АБС, который периодически экспортирует ставки в файлы и загружает на SFTP.

#### Диаграммы C4

Диаграммы приложены в файле `C4_rate_transfer.drawio`:
- Страница 1: C4 Context — контекст взаимодействия систем для передачи ставок
- Страница 2: C4 Component — детализация компонентов Rate Management Service, системы кол-центра и SFTP-интеграции

#### Логика принятия решений

1. **Почему REST API для кол-центра банка, а не файловый обмен:**
   - Система кол-центра — микросервисная (Java Spring Boot), легко интегрируется через REST API.
   - API обеспечивает получение ставок в реальном времени — оператор всегда видит актуальные данные.
   - В команде АБС есть Java-специалисты, которые могут реализовать интеграцию.

2. **Почему SFTP для партнёрского кол-центра:**
   - Партнёр работает во внешней системе и не может делать API-вызовы к банку.
   - Партнёр готов получать данные в виде файлов.
   - SFTP — стандартный безопасный протокол для файлового обмена, не требует специфической инфраструктуры.

3. **Почему Rate Export Service как отдельный компонент:**
   - Разделение ответственности: Rate Management Service управляет ставками и предоставляет API, Rate Export Service отвечает за формирование файлов и загрузку на SFTP.
   - Export Service может работать по расписанию независимо от основного API.
   - Легко расширяется при появлении новых внешних потребителей файлов.

### **Альтернативы**

| № | Альтернатива | Причина отказа |
|---|-------------|----------------|
| 1 | **Отправка ставок партнёру по email** вместо SFTP | Ненадёжно, сложно автоматизировать, нет гарантии доставки и актуальности. Email не подходит для регулярного автоматического обмена данными. |
| 2 | **Прямой доступ кол-центра к БД АБС** для получения ставок | БД АБС перегружена (из NFR задания 2). Прямой доступ внешних систем к БД — антипаттерн, создаёт жёсткую связанность. |
| 3 | **Kafka для передачи ставок в кол-центр** вместо REST API | Избыточно для данного кейса — ставки обновляются раз в день, нет потока событий. REST API проще и достаточен. Kafka уже используется для заявок (Task 3). |
| 4 | **Предоставление REST API партнёру** вместо SFTP | Партнёр явно указал, что не может делать API-вызовы. Их система не поддерживает онлайн-интеграцию с банком. |
| 5 | **Вручную отправлять файлы партнёру** (как сейчас скрипты) | Не масштабируется, ручной труд, риск ошибок и задержек. Автоматизация через SFTP надёжнее. |

**Недостатки, ограничения, риски**

| № | Категория | Описание | Митигация |
|---|----------|----------|-----------|
| 1 | **Ограничение** | Партнёрский кол-центр получает ставки с задержкой (файловый обмен не real-time). | Расписание экспорта настраивается: можно делать несколько раз в день. Ставки обновляются ежедневно — задержка в часы приемлема. |
| 2 | **Риск** | SFTP-сервер — новый компонент инфраструктуры, требует настройки безопасности и мониторинга. | Использовать OpenSSH SFTP (стандарт). Аутентификация по SSH-ключам. Мониторинг доставки файлов. |
| 3 | **Риск** | Система кол-центра — платформа подрядчика. Новый модуль отображения ставок может конфликтовать с обновлениями подрядчика. | Модуль реализуется как отдельный микросервис — изолирован от ядра платформы. Согласование с подрядчиком. |
| 4 | **Недостаток** | Дополнительная нагрузка на Rate Management Service API от запросов кол-центра (200 операторов). | Кеширование ответов API (TTL = время обновления ставок). Ставки обновляются раз в день — кеш эффективен. |
| 5 | **Ограничение** | Партнёр должен самостоятельно настроить получение файлов с SFTP. Банк предоставляет только доступы и документацию. | Подготовить документацию по формату файлов и протоколу доступа. Провести совместное тестирование. |
