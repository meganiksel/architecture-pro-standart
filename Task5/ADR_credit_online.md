### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| NFR-01 | **Доступность 24/7, SLA 99,9%.** Онлайн-подача заявок на кредит должна быть доступна круглосуточно. Кредитный конвейер и система скоринга уже поддерживают работу 24/7. |
| NFR-02 | **Отказоустойчивость.** Новые сервисы развёрнуты в обоих ЦОД. Kafka обеспечивает гарантированную доставку сообщений при сбоях. |
| NFR-03 | **Производительность.** Отклик по операциям — миллисекунды. Мгновенное решение по скорингу на сайте — не более нескольких секунд (вызов скоринга + запрос к БКИ). |
| NFR-04 | **Защита системы скоринга от перегрузки.** Предрасчёт скорингов для предодобренных предложений выполняется только в нерабочие часы. В рабочие часы скоринг используется только для текущих заявок. |
| NFR-05 | **Обход суточного пакетного обмена АБС ↔ Кредитный конвейер.** Заявки из онлайн-каналов передаются в Кредитный конвейер через Kafka напрямую, минуя суточный батч через БД. Это позволяет обрабатывать предодобренные заявки день-в-день. |
| NFR-06 | **Горизонтальное масштабирование.** Новые сервисы, Кредитный конвейер и система скоринга поддерживают горизонтальное масштабирование. АБС — только вертикально. |
| NFR-07 | **Шифрование данных.** TLS/HTTPS для всех каналов. Паспортные данные — чувствительная информация, требуется шифрование при хранении и передаче. |
| NFR-08 | **Омниканальность.** Заявка, созданная в одном канале (сайт, ИБ), должна быть доступна в другом (отделение). Единый идентификатор заявки через все системы. |
| NFR-09 | **Использование существующих технологий.** .NET Core для сервисов ИБ, Java Spring Boot для сервисов на стороне Кредитного конвейера/АБС. Kafka для асинхронного взаимодействия (уже внедрён в MVP). |
| NFR-10 | **Минимальная нагрузка на АБС.** Онлайн-заявки не должны создавать дополнительную нагрузку на БД АБС. Взаимодействие через Kafka и API сервисов. |

### **Решение**

#### Общий подход

Решение расширяет архитектуру MVP (Task 3) и добавляет кредитный процесс. Ключевые архитектурные решения:

1. **Credit Application Service** (новый микросервис в ИБ) — управляет жизненным циклом кредитных заявок из онлайн-каналов.
2. **Credit Integration Service** (новый в контуре Кредитного конвейера) — принимает заявки из Kafka напрямую, минуя суточный пакетный обмен через БД АБС.
3. **Pre-Approval Service** (новый) — рассчитывает предодобренные предложения в нерабочие часы, не нагружая скоринг в пиковые периоды.
4. **Scoring Gateway** (новый) — единая точка доступа к системе скоринга, управляет маршрутизацией запросов (онлайн-скоринг с сайта, повторный скоринг, предрасчёт).
5. **Site Credit Module** (доработка сайта) — форма заявки и интеграция со скорингом для мгновенных решений.

Диаграммы приложены в файле `C4_credit_online.drawio`:

#### Логика принятия решений

1. **Почему Kafka для передачи заявок в Кредитный конвейер (а не суточный батч):**
   - Суточный обмен через БД АБС ↔ Кредитный конвейер ускорить невозможно (техническое ограничение).
   - Бизнес требует обработку предодобренных заявок в тот же день.
   - Kafka уже внедрён в MVP (Task 3) — используем существующую инфраструктуру.
   - Credit Integration Service принимает заявки из Kafka и создаёт их непосредственно в Кредитном конвейере.

2. **Почему Pre-Approval Service работает в нерабочие часы:**
   - Система скоринга испытывает повышенную нагрузку в рабочие часы.
   - Предрасчёт скорингов — фоновая задача, не критична по времени.
   - Ночной/выходной батч позволяет обработать большое количество клиентов без влияния на операционную работу.

3. **Почему Scoring Gateway как отдельный компонент:**
   - Единая точка для управления нагрузкой на скоринг.
   - Throttling и rate limiting для защиты скоринга.
   - Кеширование результатов скоринга (один клиент не скорится повторно за короткий период).
   - Маршрутизация: разные источники запросов (сайт, предрасчёт, повторный скоринг) — разные приоритеты.

4. **Почему мгновенный скоринг на сайте через Scoring Gateway (а не через Кредитный конвейер):**
   - Клиент ожидает мгновенное решение при заполнении паспортных данных.
   - Кредитный конвейер — система обработки заявок менеджерами, не предназначена для синхронных онлайн-ответов.
   - Для нового клиента с сайта используются только данные БКИ (онлайн REST API) — скоринг может работать быстро.

5. **Почему единый идентификатор заявки (омниканальность):**
   - Клиент мог получить предодобрение на сайте → приходит в отделение → менеджер продолжает ту же заявку.
   - Credit Integration Service генерирует UUID заявки, который транслируется между каналами.
   - АБС получает информацию о существующей заявке через API Credit Integration Service.

### **Альтернативы**

| № | Альтернатива | Причина отказа |
|---|-------------|----------------|
| 1 | **Ускорить суточный батч АБС ↔ Кредитный конвейер** до нескольких раз в день | Техническое ограничение: ускорить обмен через БД невозможно (явно указано в задании). Kafka решает проблему, не затрагивая существующую интеграцию. |
| 2 | **Предрасчёт скорингов в рабочие часы** с low-priority очередью | Система скоринга перегружена в рабочие часы — даже low-priority запросы создадут дополнительную нагрузку. Ночной батч полностью исключает конфликт. |
| 3 | **Прямой вызов системы скоринга с сайта** без Scoring Gateway | Нет контроля нагрузки и кеширования. Риск перегрузки скоринга при маркетинговых кампаниях. Gateway обеспечивает throttling и единый интерфейс. |
| 4 | **Хранение предодобренных предложений в АБС** вместо отдельного кеша | АБС перегружена, дополнительные записи/чтения увеличат нагрузку. Отдельный кеш (Pre-Approval Cache) быстрее и не нагружает АБС. |
| 5 | **Реализация кредитной заявки как доработка монолита ИБ** | Те же причины, что в Task 3: устаревший стек, зависимость от подрядчика, несовместимость с Kafka. Микросервисный подход уже выбран и внедрён. |
| 6 | **Отдельный Credit Pipeline для онлайн-заявок** вместо интеграции с существующим | Дублирование логики обработки заявок. Кредитный конвейер уже поддерживает горизонтальное масштабирование. Лучше интегрировать через Kafka. |

**Недостатки, ограничения, риски**

| № | Категория | Описание | Митигация |
|---|----------|----------|-----------|
| 1 | **Ограничение** | Предодобренные предложения рассчитываются с задержкой (ночной батч). Клиент видит данные, актуальные на предыдущий день. | Ставки меняются ежедневно — задержка в один день приемлема. При подаче заявки скоринг проводится повторно для актуальной оценки. |
| 2 | **Ограничение** | Новые клиенты с сайта получают решение только на основе БКИ (нет внутренних данных банка). Точность скоринга ниже. | Это осознанное ограничение: для нового клиента нет данных в АБС. При визите в отделение проводится полный скоринг с учётом всех данных. |
| 3 | **Риск** | Кредитный конвейер — коробочное решение на Camunda. Интеграция через Credit Integration Service может быть сложной. | В IT-отделе есть специалисты по Camunda. Credit Integration Service изолирует новую логику от ядра конвейера. Подрядчик привлекается при необходимости. |
| 4 | **Риск** | Scoring Gateway — единая точка отказа для всех скоринговых запросов. | Развёртывание в нескольких экземплярах с балансировкой. Health checks и автоматический failover. Circuit breaker для защиты при сбоях скоринга. |
| 5 | **Ограничение** | Суточный пакетный обмен АБС ↔ Кредитный конвейер продолжает работать параллельно с Kafka. Два канала доставки заявок. | Credit Integration Service обеспечивает дедупликацию: заявки из Kafka помечаются как «онлайн» и не дублируются при суточном батче. |
| 6 | **Риск** | При большом количестве клиентов предрасчёт скорингов в ночное окно может не уложиться по времени. | Горизонтальное масштабирование системы скоринга и Pre-Approval Service. Приоритизация: скорить в первую очередь активных клиентов. |
| 7 | **Ограничение** | Клиент обязан посетить отделение для идентификации (KYC) и получения кредита. Полностью онлайн-кредит невозможен для новых клиентов. | Регуляторное ограничение. Для существующих клиентов ИБ процесс максимально автоматизирован. |
| 8 | **Риск** | Паспортные данные с сайта — высокочувствительная информация. Утечка может привести к репутационным и финансовым потерям. | TLS/HTTPS обязателен. Шифрование данных at rest. Минимизация хранения — паспортные данные передаются в скоринг и не сохраняются на сайте. Аудит доступа. |
